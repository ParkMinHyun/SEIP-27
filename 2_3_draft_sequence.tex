\subsection{The Draft Sequence on the Capture Timeout Path}
\label{sec:draft-background}

A shutter event requests multiple image buffers and their metadata.
As Figure~\ref{fig:draft-pipeline} shows, the camera framework uses these inputs along two complementary paths.
The final path composes multiple frames, applies single-frame processing, and stores the final image.
The baseline Draft Sequence instead selects and stores a representative frame as the Draft image for early display.
When final processing completes, its result replaces the Draft image; if it fails or is interrupted, the Draft image remains available for recovery.
The Draft Sequence is therefore both user-facing and reliability-critical.

\smallskip\noindent\textbf{Increasing Draft workload.}
As post-processing grew more compute- and memory-intensive, selected modes began deferring final composition until the camera application entered the background.
The longer-lived Draft made visual consistency with the final image more important.
The deployed Draft Sequence consequently incorporated a lightweight single-frame stage before mandatory encoding and storage.
We have implemented a lightweight multi-frame counterpart to the final compositing stage for Portrait mode, where the Draft--final difference is particularly pronounced.
However, safely enabling this additional Draft processing in production remains difficult under the existing safeguards.

\smallskip\noindent\textbf{Capture-ordered Draft execution.}
The framework executes Draft Sequences on a single worker thread in capture order.
This production backpressure preserves result publication and storage order while limiting active Draft compute and working memory to one sequence.
Multiple workers would require buffering early-finishing results and would make Draft workloads compete concurrently with preview and capture processing.
