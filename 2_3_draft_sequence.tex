\subsection{Draft Processing and Production Constraints}
\label{sec:draft-background}

Each capture is handled by both the Draft Sequence and the post-processing pipeline, as illustrated in Figure~\ref{fig:draft-pipeline}.
The post-processing pipeline produces and stores the final image.
Introduced alongside this pipeline, the Draft Sequence publishes the Draft image as an early user-visible result.
If the Draft image has already been published, the final image replaces it when post-processing completes; if post-processing fails or is interrupted, the Draft image remains available for recovery.
The Draft Sequence is therefore both user-facing and reliability-critical.

\smallskip\noindent\textbf{Capture-ordered Draft execution.}
The single-worker design described in Section~\ref{sec:parallel-capture} preserves capture-order publication and storage while limiting active Draft compute and working memory to a single sequence.
With multiple workers, later sequences could finish first, requiring their results to remain buffered until earlier sequences complete; concurrent Draft processing would also contend with preview and capture workloads for compute and memory.

\smallskip\noindent\textbf{Increasing Draft workload.}
The initial Draft Sequence selected a representative frame and proceeded directly to required encoding and storage.
To reduce visible differences between the Draft image and final image, the deployed Draft Sequence later incorporated a lightweight single-frame processing stage.
As post-processing became more compute- and memory-intensive, selected shot modes began deferring the full pipeline until the camera application entered the background, leaving the Draft image visible longer and making these differences more noticeable.
For Portrait mode, in which the visual gap between the Draft image and final image is particularly pronounced, we have implemented a lightweight multi-frame Draft stage across all supported device models to approximate the deferred post-processing result.
Safely enabling this stage in production requires controlling its additional workload on the Capture Timeout path.
