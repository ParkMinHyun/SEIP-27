\subsection{Draft Processing and Production Constraints}
\label{sec:draft-background}

Within each capture, the Draft Sequence and post-processing pipeline produce the Draft and final images, respectively.
If a Draft image has been published, the final image replaces it when post-processing completes; if post-processing fails or is interrupted, the Draft image remains available for recovery.
The Draft Sequence supports both user-perceived responsiveness and capture reliability.
Figure~\ref{fig:draft-pipeline} illustrates this workflow.

\smallskip\noindent\textbf{Single-worker Draft execution.}
Like post-processing, the Draft path uses single worker to preserve capture-order publication and limit active processing to one sequence.
Concurrent Draft execution would require buffering out-of-order results to preserve order and would overlap workloads across captures, raising peak compute and memory demand and degrading performance as active sequences increase.

\smallskip\noindent\textbf{Increasing Draft processing workload.}
Initially, the Draft Sequence selected a representative frame and proceeded directly through the required encoding and saving stages.
A later release added a lightweight single-frame stage to reduce visible differences between Draft and final images.
As post-processing grew more compute- and memory-intensive, selected shot modes began deferring this full pipeline until the camera application entered the background, prolonging Draft visibility and making these differences more noticeable.
Because this gap is pronounced in Portrait mode, we developed a lightweight multi-frame Draft stage as a production candidate for supported models to approximate the deferred result.
Under parallel capture, however, this stage can grow the Draft-worker backlog and increase Capture Timeout risk.
