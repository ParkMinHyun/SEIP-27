\subsection{Draft Processing and Production Constraints}
\label{sec:draft-background}

Within each capture, the Draft Sequence and post-processing pipeline produce the Draft and final images, respectively.
If a Draft image has been published, the final image replaces it when post-processing completes; if post-processing fails or is interrupted, the Draft image remains available for recovery.
The Draft Sequence supports both user-perceived responsiveness and capture reliability.
Figure~\ref{fig:draft-pipeline} illustrates this workflow.

\smallskip\noindent\textbf{Single-worker Draft execution.}
The Draft path, like the post-processing path, deliberately uses a single worker to preserve capture-order publication and limit active processing to one sequence.
Concurrent Draft execution would require buffering out-of-order results to preserve this order and would overlap Draft workloads across captures, thereby raising peak compute and memory demand and degrading camera performance as more sequences become active.

\smallskip\noindent\textbf{Increasing Draft processing workload.}
The initial Draft Sequence selected a representative frame and proceeded directly to required encoding and storage.
To reduce visible differences between the Draft image and final image, the deployed Draft Sequence later incorporated a lightweight single-frame processing stage.
As post-processing became more compute- and memory-intensive, selected shot modes began deferring the full post-processing pipeline until the camera application entered the background, leaving the Draft image visible longer and making these differences more noticeable.
For Portrait mode, where this gap is pronounced, we have implemented a lightweight multi-frame Draft stage on all supported models to approximate the deferred post-processing result.
Under parallel capture, however, this added multi-frame processing can grow the backlog of Draft Sequences waiting for the Draft worker and increase Capture Timeout risk.
