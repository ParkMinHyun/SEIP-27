Table~\ref{tab:bug_main} demonstrates that $\ourtool$ is also effective for bug finding. To identify bug-triggering test cases, during the experiments in Table~\ref{tab:coverage_main} and Figure~\ref{fig:long_coverage}, we first collected all test cases for which KLEE generated an error report as a *.err file within the total testing budget (i.e., 12h $\times$ 5 repeats). Using the collected test cases, we re-executed them on the original binaries to filter out the false positives. For the test cases that triggered actual bugs, we report the program version in which the bug occurred, the type of bug, and the bug-triggering code line. For each bug, ($\signO$) indicates that the corresponding baseline detected the crash. If bugs of the same type occurred at the same code line across consecutive versions, we counted them as a single bug and reported the version as the range from the first observed version to the last observed version.

In total, $\ourtool$ detected nine unique bugs across six programs. This is 125\% more than $\default$ and $\rerun$, which found four bugs, and 80\% more than $\seeding$ and $\naive$, which detected five bugs. Remarkably, $\ourtool$ did not miss any bugs detected by the baselines, while detecting three bugs that none of the baselines found. We reported all bugs found by $\ourtool$ to the program developers. The bugs found in $\diff$, $\gawk$, $\ls$, and $\sqlite$ have been fixed, and the bugs found in $\patch$ have been confirmed.
