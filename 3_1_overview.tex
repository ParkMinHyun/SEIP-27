\subsection{Overview}\label{sec:overview}

\begin{figure}
    \centering
    \includegraphics[width=\columnwidth]{figures/overview3.pdf}
    \caption{The overview of $\ourtool$.}
    \label{fig:overview}
\end{figure}
Figure~\ref{fig:overview} presents an overview of $\ourtool$. During symbolic execution, $\ourtool$ accumulates query data and refines it for reuse across subsequent software releases. To achieve this, $\ourtool$ first refines query data through three stages: $\sample$, $\extract$, and $\generalize$. After symbolic execution completes, satisfiable and unsatisfiable query data are processed separately. For satisfiable query data, the $\sample$ stage resamples satisfying assignments to generate diverse test cases from the same path condition. For unsatisfiable query data, the $\extract$ stage identifies a minimal infeasible core from the path condition, allowing later path conditions containing the same core to be classified as unsatisfiable without invoking the SMT solver.
The $\generalize$ stage then abstracts both satisfiable and unsatisfiable query data into generalized representations, allowing multiple related query data entries to be merged into a single reusable entry. 

The refined query data are subsequently accumulated and reused during symbolic execution of later software releases. For each newly generated path condition, $\ourtool$ prioritizes the most promising query data for reuse and determines whether they can be safely reused without invoking the SMT solver. This iterative process progressively accumulates refined query data, reducing redundant SMT solving and improving the efficiency of symbolic execution across software releases.

%Figure~\ref{fig:overview} illustrates the process by which $\ourtool$ refines symbolic execution results and delivers them to subsequent executions. Given query-solving results from a previous execution, $\ourtool$ refines them through four stages---$\sample$, $\extract$, $\generalize$, and $\compare$---for efficient reuse. Once symbolic execution completes and returns query results, $\ourtool$ processes unsatisfiable and satisfiable query data separately, each through an independent stage. In the $\sample$ stage, $\ourtool$ constructs new assignments for satisfiable query path conditions that can be satisfied by more than one assignment. The re-sampled assignments allow $\ourtool$ to generate diverse test cases from a single path condition. For unsatisfiable query data, the $\extract$ stage identifies a minimal set of conditions that makes each query’s path condition unsatisfiable. Using the extracted core condition set, any path condition containing this set is directly calculated as unsatisfiable.  

%After refining the query results, in the $\generalize$ stage, $\ourtool$ transforms path conditions in the results by abstracting their variables and values. This allows multiple queries to be represented by a single data entry. Lastly, with the well-structured dataset, $\ourtool$ loads them to effectively reduce constraint-solving costs when testing new releases. In the $\compare$ stage, $\ourtool$ computes the path-condition-based similarity for each loaded data entry and tries highly similar entries first when checking the feasibility of a query. This four-stage process repeats for each execution on a new release, gradually accumulating data and making testing increasingly efficient for subsequent releases.
