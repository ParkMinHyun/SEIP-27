\section{Introduction}\label{sec:introduction}

Applying lightweight Bokeh to the temporary image shown after a portrait
capture appears to be a local enhancement. In a commercial mobile-camera
framework, it became a release-blocking systems problem. Modern mobile cameras form a
final photograph from multiple frames using compute- and memory-intensive
processing. To preserve preview responsiveness, selected capture modes defer
this full composition until the camera application moves to the background.
In the meantime, a deadline-critical \emph{Draft Sequence} produces an early
image so that the user can inspect the new capture and the system retains a
recovery image. As final processing has become more sophisticated, however,
the Draft remains visible for longer and can differ sharply from the final
result. The mismatch is particularly conspicuous in portrait mode: the early
image may lack the background blur and tonal treatment that define the final
portrait. Lightweight multi-frame Draft Bokeh and Filter are therefore not
merely cosmetic additions; they are the mechanism that makes deferred portrait
processing acceptable to users.

Placing these effects on the Draft path is difficult because the path is part
of a fixed \emph{Capture Timeout} deadline that also includes mandatory
encoding and result publication. Capture Timeout compliance is one of this
product line's highest-priority camera reliability KPIs and a hard release gate:
any reproducible violation blocks launch until the responsible feature is fixed,
restricted, or removed.
Moreover, the deadline is not an isolated per-image compute budget. Parallel
capture allows the hardware
abstraction layer (HAL) to become ready for another shot before the preceding
Draft Sequence has finished, whereas all Draft Sequences execute on a single
worker. A later capture can thus spend much of its deadline waiting behind
earlier work. One tail-latency spike propagates through the burst, and an
operation that is safe in isolation can time out after several consecutive
shots.

This cross-shot interference prevented Draft Bokeh from shipping in two
consecutive flagship generations.

The deployed safeguard does not address this failure mechanism. After parallel
capture was already in production, a Filter-related timeout on a lower-tier
device led the shared framework to disable optional Draft processing at
thermal level~4 and above. The rule was effective as an incident-containment
measure, but it is a poor feature-enablement policy. Below level~4, it admits
Bokeh even when accumulated queueing has exhausted the remaining deadline. At
level~4 and above, it rejects all optional work even when the Draft worker is
idle and the current capture has ample slack. Resolution, memory pressure,
workload composition, and burst history can produce different outcomes at the
same thermal level. Consequently, moving the threshold merely exchanges one
failure for another: lowering it discards more useful processing, whereas
raising it accepts more timeout risk.

We address the problem with a context-aware controller that coordinates two
control points. \emph{Remaining-sequence admission} chooses which optional
stages to execute by comparing a context-conditioned upper bound for all
remaining work, including the mandatory tail, with the current capture's
remaining budget. Admission protects the capture already in progress, but it
cannot recover time that the capture has already lost in the queue.
\emph{Capture-availability pacing} therefore delays delivery of the next
\texttt{captureAvailable} callback by the predicted budget deficit, allowing
the single Draft worker to make progress before another capture---and another
deadline---begins. Both decisions reuse an online, tail-calibrated execution-
time model. Admission controls the service demand of the current capture;
pacing controls the arrival of subsequent captures.

In a controlled stress evaluation comprising three 30-shot, 12MP portrait
bursts at thermal levels~4--6---the entire region in which the deployed guard
would disable optional Draft work---our prototype completed all 90 captures
without a timeout while applying both Draft Bokeh and Filter to 63 captures
(70.0\%). The mean callback delay per burst ranged from 0.85 to 1.15 seconds,
with a maximum of 1.50 seconds. These results quantify the central trade-off:
the controller retains substantial Draft functionality under conditions where
the static policy provides none, while exposing the responsiveness cost needed
to preserve the deadline.

\smallskip
\noindent\textbf{Contributions.}
This paper makes the following contributions:
\begin{itemize}[leftmargin=*]
  \item \textbf{An industrial characterization of cross-shot deadline
  interference.} Using portrait bursts across thermal levels, resolutions,
  and memory conditions, we show why a shared thermal threshold is both
  under-protective and over-restrictive.

  \item \textbf{A coordinated runtime controller.} We combine
  remaining-sequence admission for the current capture with adaptive
  capture-availability pacing for subsequent captures, driven by a shared
  online model of tail execution time.

  \item \textbf{An evaluation in a commercial Android camera framework.}
  We measure deadline safety, Draft feature availability, prediction quality,
  and callback-level pacing cost under the runtime conditions that previously
  prevented product enablement.
\end{itemize}
