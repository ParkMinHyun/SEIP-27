\section{Introduction}\label{sec:introduction}
Symbolic execution~\cite{Cadar2008,Godefroid2005,crest,angr} is a representative white-box testing technique that automatically generates test cases to achieve high code coverage and detect hidden bugs. Its key idea is to replace program inputs with symbolic variables and systematically explore program paths by reasoning about symbolic path conditions using an SMT solver~\cite{Ganesh07,z32008,brummayer2009boolector,barbosa2022cvc5}. During execution, symbolic execution repeatedly selects and executes symbolic program states, and whenever a state reaches a program exit, it invokes the SMT solver to determine whether the accumulated path condition is satisfiable and, if so, obtains a corresponding test case. Although symbolic execution has demonstrated remarkable effectiveness in automated software testing, its practical applicability is still hindered by two long-standing challenges: the high cost of SMT solving and the state-explosion problem. To address these challenges, existing techniques have proposed a wide range of optimizations, including SMT-solving optimizations~\cite{shuai2024partial,trabish2021address,shuai2021type,Kapus2020}, search heuristics~\cite{yao2025empc,lee2025topseed,featmaker,learch}, and state-pruning techniques~\cite{du2026fense,yi2024compatible,homi,Yi2018}.

Despite these advances, existing symbolic execution techniques are not designed to effectively test successive releases of the same software, even though adjacent releases typically share most of their code. As we show in Section~\ref{sec:limitation}, 89.8\% of the path conditions explored in a later release have already been solved while testing earlier releases. Re-solving these path conditions consumes approximately 70\% of the total testing budget, and the overhead accumulates as software continues to evolve.

A straightforward solution is to store the results of solved path conditions from earlier releases and reuse them during symbolic execution of later releases. Whenever a newly generated path condition matches a stored result, symbolic execution can immediately reuse it instead of invoking the SMT solver. However, such a naive query-reuse approach suffers from three fundamental limitations. First, storing every solved path condition and its corresponding result incurs substantial storage overhead. Second, searching through a large collection of stored results introduces considerable matching overhead. Finally, even when a matching result is found, it often reconstructs test cases that have already been generated in previous releases, reducing testing diversity and limiting the exploration of new program behaviors.

We present $\ourtool$, a practical technique that refines previously solved query results into reusable query data, enabling symbolic execution to effectively test successive releases of the same software. To this end, $\ourtool$ first processes satisfiable and unsatisfiable query data separately. For satisfiable query data, it resamples satisfying assignments without additional SMT solving, avoiding regenerating the same test cases across releases. For unsatisfiable query data, it extracts minimal infeasible cores so that later path conditions containing the same core can be classified as unsatisfiable without invoking the SMT solver. The refined query data are then generalized by abstracting symbolic variables and normalizing concrete values, allowing multiple semantically equivalent query data entries to be represented by a single reusable entry. During symbolic execution of successive releases, $\ourtool$ prioritizes the most promising query data according to their similarity to newly generated path conditions and reuses matched query-data entries through a more general matching criterion than exact path-condition equality. Consequently, as reusable query data accumulate across software releases, $\ourtool$ progressively reduces SMT solver invocations by reusing previously refined query-data entries to determine satisfiability and reconstruct satisfying assignments.

Our experimental evaluation demonstrates that $\ourtool$ improves symbolic execution across successive software releases in terms of both branch coverage and bug-finding ability. We implemented $\ourtool$ on top of KLEE~\cite{Cadar2008} and evaluated it on consecutive releases of 15 open-source C programs. We compared $\ourtool$ against four baselines: original KLEE, the naive query-reuse baseline, a rerun baseline that re-executes test cases generated from previous releases, and a seed-based baseline that uses those test cases as seeds for testing later releases. For branch coverage, $\ourtool$ covered 28.5\% more branches than the baselines on the second release and 30.9\% more branches on the third release, showing that its benefits grow as reusable query data accumulate across releases. In terms of bug finding, $\ourtool$ detected eight crashes across three consecutive releases, including three missed by all baselines. We further demonstrate that $\ourtool$ is complementary to existing techniques by integrating it with three state-of-the-art symbolic execution techniques~\cite{yao2025empc,lee2025topseed,klee_aaqc}, improving their effectiveness. Moreover, we demonstrate the generality of $\ourtool$ by applying it to two concolic testing tools~\cite{poeplau2020symbolic,chameleon}, confirming that its design is independent of the underlying symbolic execution engine.

\myparagraph{\bf{Contributions}}
We make the following contributions:
\begin{itemize}[leftmargin=*]
\item {\bf New problem.} We show that symbolic execution repeatedly re-solves path conditions across successive software releases, wasting substantial SMT-solving time despite the high code similarity between adjacent releases.
\item {\bf Practical technique.} We present $\ourtool$, a practical technique that progressively refines previously solved query results into reusable query data, enabling symbolic execution to reduce redundant SMT solver invocations while avoiding regenerating the same test cases across successive releases.
\item {\bf Comprehensive evaluation.} We evaluate $\ourtool$ on consecutive releases of 15 open-source C programs against four baselines. We further demonstrate that $\ourtool$ complements existing symbolic execution techniques and generalizes to concolic testing tools. Our implementation is publicly available at: \url{https://github.com/unknownicse27/reuse}.
\end{itemize}
